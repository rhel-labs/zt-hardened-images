= TLS Reverse Proxy

Applications in production typically sit behind a reverse proxy that handles TLS.
This module runs Flask and Caddy in a pod, observes the certificate trust problem, and resolves it by embedding the CA into a custom `hi/curl` image.

== Create a Pod

Pods in Podman share a network namespace.
Caddy can reach Flask on `localhost:8080` without any explicit container-to-container networking.

[source,bash,role=execute]
----
podman pod create --name rhhi-pod -p 8080:8080 -p 8443:8443
----

== Start Flask in the Pod

[source,bash,role=execute]
----
podman run -d --pod rhhi-pod --name rhhi-flask rhhi-demo:v1
----

== Start Caddy in the Pod

The Caddyfile is pre-staged at `~/webserver/Caddyfile`:

[source,bash,role=execute]
----
cat ~/webserver/Caddyfile
----

It configures Caddy to listen on `:8443` with `tls internal` and reverse proxy to `localhost:8080`.

[source,bash,role=execute]
----
podman run -d --pod rhhi-pod --name rhhi-caddy \
  -v ~/webserver/Caddyfile:/etc/caddy/Caddyfile:Z \
  registry.access.redhat.com/hi/caddy:latest
----

== Observe the Certificate Problem

Open the *Caddy (TLS)* tab.
You'll see a certificate warning — Caddy generated its own CA and issued a certificate signed by it.
Your browser doesn't trust that CA.

`curl` from the terminal fails the same way:

[source,bash,subs=attributes+,role=execute]
----
curl https://caddy-{guid}.{domain}/
----

Using `-k` bypasses verification and the connection works — but that defeats TLS entirely.
The right fix is to trust the CA, not skip verification.

== Extract the CA Certificate

Caddy writes its internal CA key and certificate to `/data/caddy/pki/authorities/local/` inside the running container.
Extract both files and combine them into a single PEM that the `trust` command expects:

[source,bash,role=execute]
----
podman cp rhhi-caddy:/data/caddy/pki/authorities/local/root.key .
podman cp rhhi-caddy:/data/caddy/pki/authorities/local/root.crt .
cat root.key root.crt > ca.pem
----

[source,bash,role=execute]
----
openssl x509 -in root.crt -noout -subject -dates
----

== Build a Trusted curl Image

`hi/curl:latest-builder` includes the `trust` command.
Use it in a builder stage to add the CA, then copy only the resulting trust store into the lean runtime.

[source,bash,role=execute]
----
cat > ~/Containerfile.curl << 'EOF'
FROM registry.access.redhat.com/hi/curl:latest-builder AS build
COPY ca.pem /tmp/ca.pem
USER root
RUN trust anchor /tmp/ca.pem
USER ${CONTAINER_DEFAULT_USER}

FROM registry.access.redhat.com/hi/curl:latest
COPY --from=build /etc/pki/ca-trust/extracted /etc/pki/ca-trust/extracted
EOF
----

[source,bash,role=execute]
----
podman build -t rhhi-curl:trusted -f ~/Containerfile.curl ~
----

== Verify TLS

The standard `hi/curl` image doesn't trust the Caddy CA:

[source,bash,subs=attributes+,role=execute]
----
podman run --rm registry.access.redhat.com/hi/curl:latest https://caddy-{guid}.{domain}/
----

The custom image does:

[source,bash,subs=attributes+,role=execute]
----
podman run --rm rhhi-curl:trusted https://caddy-{guid}.{domain}/
----

The request succeeds without `-k`.
The CA is trusted at the image level — no runtime flags, no verification skipped.

== Wrap Up

Over these four modules, the same Flask application ran on four different bases:

[options="header"]
|===
| Image | Base | Key Characteristic
| `rhhi-demo:ubi`     | `ubi9/python-312`     | General-purpose, full package set
| `rhhi-demo:builder` | `hi/python:3.12-builder` | Builder tools present — not for production
| `rhhi-demo:v1`      | `hi/python:3.12`      | Curated runtime, minimal CVE surface
| `rhhi-demo:fips`    | `hi/python:3.12-fips` | FIPS enforcement at the OpenSSL level
|===

Red Hat Hardened Images are production-focused variants with a consistent catalog, predictable rebuild cadence, and variant support (default, builder, fips) designed to match the full container lifecycle.
