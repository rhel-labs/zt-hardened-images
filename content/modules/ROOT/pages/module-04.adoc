= TLS Reverse Proxy

Applications in production typically sit behind a reverse proxy that handles TLS.
Organizations will often have internal certificates or authorities (CAs) that need to be used to provide appropriate chains of trust for signing.
This module runs Flask and Caddy in a pod, and relies on Caddy’s built in TLS support to simulate a CA.

== Create a Pod

Pods in Podman share a network namespace, which makes them a useful tool for running multiple independent containers as a single application.
Caddy can reach Flask on `localhost:8080` without any explicit container-to-container networking.
We won't be exposing that port outside the pod, just the TLS port Caddy will use.

[source,bash,role=execute]
----
podman pod create --name rhhi-pod -p 8443:8443
----

== Start the application in the Pod

We'll start the demo application in the pod first.
[source,bash,role=execute]
----
podman run -d --pod rhhi-pod --name rhhi-flask rhhi-demo:v1
----

== Start Caddy in the Pod

The Caddyfile is pre-staged at `~/webserver/Caddyfile`:

[source,bash,role=execute]
----
cat ~/webserver/Caddyfile
----
[source,dockerfile]
----
include::example$setup-files/flask/Caddyfile[]
----

It configures Caddy to listen on the lab's external hostname with `tls internal` and reverse proxy to `localhost:8080`.
Caddy will generate a certificate from its internal CA for that hostname.
We can just pass our custom Caddyfile as a volume to the image, but for a more robust production use case, it may be better to create a fully custom image.

[source,bash,role=execute]
----
podman run -d --pod rhhi-pod --name rhhi-caddy \
  -v ~/webserver/Caddyfile:/etc/caddy/Caddyfile:Z \
  registry.access.redhat.com/hi/caddy:latest
----

== Observe the Certificate Problem

Navigate to link:https://caddy-{guid}.{domain}/[https://caddy-{guid}.{domain}/^] in your browser.
You'll see a certificate warning, your browser doesn't trust that CA. You could examine and proceed, but during pipeline testing that may not be possible.

`curl` from the terminal fails the same way:

[source,bash,subs=attributes+,role=execute]
----
curl https://caddy-{guid}.{domain}/
----

Using `-k` bypasses verification and the connection works — but that defeats TLS entirely.
The right fix is to trust the CA, not skip verification.

== Extract the CA Certificate

Caddy writes its internal CA key and certificate to `/data/caddy/pki/authorities/local/` inside the running container.
Extract both files and combine them into a single PEM that the `trust` command expects:

[source,bash,role=execute]
----
podman cp rhhi-caddy:/data/caddy/pki/authorities/local/root.key .
podman cp rhhi-caddy:/data/caddy/pki/authorities/local/root.crt .
cat root.key root.crt > ca.pem
----
This is similar to what you'd need from your internal CA.

[source,bash,role=execute]
----
openssl x509 -in root.crt -noout -subject -dates
----

===  Build custom curl with CA bundle

Using our internal CA as part of the trusted chain, we'd need to embed it in the image for our applications or tools. We'll use the `builder` variant of the `curl` image to create a custom internal `curl` we could use to validate custom certificates in a pipeline.

This multi-stage pattern is the same we used before. This keeps all of the work of creating the new trust store in our builder and only adds the files we need to our final `curl` tool image.

[source,bash,role=execute,subs=attributes+]
----
cat > ~/Containerfile.pem << 'EOF'
FROM {hummingbird-registry}/curl:latest-builder as builder

# Copy the certificate to the image
COPY ca.pem /tmp/ #<.>

# Temporarily switch to root to add the CA certificate to the trust store
USER root
RUN trust anchor /tmp/ca.pem <.>
USER ${CONTAINER_DEFAULT_USER}


# Runtime stage:
# Copy the trust store from the builder image to the runtime image
FROM {hummingbird-registry}/curl:latest
COPY --from=builder /etc/pki/ca-trust/extracted /etc/pki/ca-trust/extracted <.>
EOF
----
<.> Copy the custom CA pem to the `build` stage
<.> Include the custom CA in our trusted root
<.> Only copy the PKI files into the final image

Now build our new tool with the internal CA trusted.
[source,bash,role=execute]
----
podman build -t rhht-curl:local-ca -f ~/Containerfile.pem ~
----

== Verify the chain with our new image

The standard `hi/curl` image doesn't trust the Caddy CA:

[source,bash,subs=attributes+,role=execute]
----
podman run --rm registry.access.redhat.com/hi/curl:latest https://caddy-{guid}.{domain}/
----

The custom image does:

[source,bash,subs=attributes+,role=execute]
----
podman run --rm rhhi-curl:local-ca https://caddy-{guid}.{domain}/
----

== Wrap Up

Over these four modules, the same Flask application ran on four different bases:

[options="header"]
|===
| Image | Base | Key Characteristic
| `rhhi-demo:ubi`     | `ubi9/python-312`     | General-purpose, full package set
| `rhhi-demo:builder` | `hi/python:3.12-builder` | Builder tools present — not for production
| `rhhi-demo:v1`      | `hi/python:3.12`      | Curated runtime, minimal CVE surface
| `rhhi-demo:fips`    | `hi/python:3.12-fips` | FIPS enforcement at the OpenSSL level
|===

Red Hat Hardened Images are production-focused variants with a consistent catalog, predictable rebuild cadence, and variant support (default, builder, fips) designed to match the full container lifecycle.
