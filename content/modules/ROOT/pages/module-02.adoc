= Multi-Stage Hardened Build

Red Hat Hardened Images for runtimes come in two variants: a `default` distroless runtime image and a `builder` variant that adds bash, dnf, and shadow-utils for the build stage.
The pattern is a multi-stage build: install dependencies in the builder, then copy only what you need into the lean runtime.

== Build the Hardened Image

`Containerfile.hardened` uses `hi/python:3.12-builder` to install Flask dependencies into a prefix directory, then copies that prefix into the distroless `hi/python:3.12` runtime.

[source,bash,role=execute]
----
podman build -t rhhi-demo:v1 -f ~/flask/Containerfile.hardened ~/flask
----

== Compare Image Sizes

[source,bash,role=execute]
----
podman images rhhi-demo
----

The hardened runtime image is significantly smaller than the UBI baseline.
Fewer bytes in the image means fewer packages, fewer files, and fewer CVE exposure points.

== Run the Hardened Image

[source,bash,role=execute]
----
podman run -d --name rhhi-hardened -p 8080:8080 rhhi-demo:v1
----

[source,bash,role=execute]
----
echo "http://flask-${GUID}.${DOMAIN}/crypto-demo"
----

Navigate to that URL.
The app behaves identically — same hashes, same form.
The difference is what's _under_ the app.

[source,bash,role=execute]
----
podman stop rhhi-hardened && podman rm rhhi-hardened
----

== CVE Comparison with Grype

Scan both images and compare:

[source,bash,role=execute]
----
grype rhhi-demo:ubi --output table 2>/dev/null | tail -20
----

[source,bash,role=execute]
----
grype rhhi-demo:v1 --output table 2>/dev/null | tail -20
----

The UBI image carries CVEs from the general-purpose packages included for broad compatibility.
The HI runtime image contains only what the application needs to run.

[NOTE]
====
CVE counts change as patches are released.
The point is the difference in exposure, not the specific numbers.
====

== Package Inventory with Syft

Generate an SBOM for the hardened image:

[source,bash,role=execute]
----
syft rhhi-demo:v1 -o spdx-json > ~/rhhi-demo-v1.spdx
----

[source,bash,role=execute]
----
jq '.packages | length' ~/rhhi-demo-v1.spdx
----

[source,bash,role=execute]
----
jq '.packages | group_by(.primaryPackagePurpose) | map({type: .[0].primaryPackagePurpose, count: length})' ~/rhhi-demo-v1.spdx
----

The SBOM shows the RPM base packages alongside the Python packages installed by pip during the build stage.

== Key Insight

The curated HI Python runtime includes only the packages needed for the interpreter to run.
No compilers, no build tools, no extra RPMs from a general-purpose package set.
The builder variant carries those tools — but only in the layer used at build time, not in the final image.
