= Red Hat Hardened Images


[[types]]
== Available types of hardened images

There are three main categories of images provided in the collection of hardened images: runtimes, services, and tools. Each of these are built in the same fashion but provide different capabilities.  Runtimes include various languages like `java`, `python`, and `dotnet` and also have `-builder` variants to allow for complex application tasks. Services include network based applications like `caddy`, `memcached`, and `haproxy` for deploying minimal versions of common services. Tools include things like `git`, `curl`, and `jq` for running a known containerized version in an isolated manner like within a CI pipeline.

== Distroless vs distribution base images

Hardened images are what's called 'distroless' images. These styles of images are targeted builds that skip certain shells and tools typically found in standard base images. Since package managers are usually part of that missing toolset, that's how the term 'distroless' was coined.

[NOTE]
====
Throughout this lab, you'll see files marked with circled numbers and a numbered list following the file. These are to help draw your attention to particular parts of the file. These are part of the source file and also visible in the terminal via `cat` or `vi`. The numbers will not be included if you copy from one of the blocks.
====

Red Hat Hardened Images for Python come in variants.
The `default` variant is distroless — no shell, no package manager, only what the interpreter needs at runtime.
The `builder` variant adds bash, dnf, and shadow-utils for dependency installation.

Starting with the default variant shows what distroless means in practice.

== Try the Default Variant

Create a new Containerfile using the default `hi/python:3.14` runtime:

[source,bash,role=execute]
----
cat > ~/flask/Containerfile.hi << 'EOF'
# Stage 1: Hardened base image
FROM {hummingbird-registry}/python:3.14 #<.>
ENV RHHI_VARIANT=runtime

# Switch to root from the non-root user to install dependencies
# By default, these install in /tmp/.local for the non-root user
USER root
RUN pip install flask

# Set the working directory in the container
WORKDIR /app

# Copy the application files to the target directory
# COPY always executes as root
COPY --chown=65532 app.py . #<.>

# Switch to the default non-root user for runtime
USER ${CONTAINER_DEFAULT_USER} #<.>

# Expose the port Flask will listen on
EXPOSE 8080

# Appropriately set the stop signal for the python interpreter executed as PID 1
STOPSIGNAL SIGINT
ENTRYPOINT ["python", "./app.py"]

EOF
----
Distroless characteristics:

<.> Change our base image to the hardened `default` image
<.> Change the user in `COPY` to the provided default UID instead of adding `appuser`
<.> Reset the USER with the provided ENV variable

Let's go ahead and build the demo application on the `default` Python variant.
[source,bash,role=execute]
----
podman build -t rhhi-demo:default -f ~/flask/Containerfile.hi ~/flask
----

.*Expected output (partial):*
----
error running container: ... executable file `/bin/sh` not found: No such file or directory
----
[NOTE]
====
This failure is deliberate to pivot to our next topic. However if you know a binary or command is available in the image, you can use the "exec form" instead of the "shell form" for `RUN` like `CMD`:

RUN ["/usr/bin/python", "-m",  "pip", "install", " flask" ]
====

What happened here? One of the changes to 'distroless' images like our new `default` is the removal of things like shells. 
The `RUN` step executes provided commands (here `pip install`) by passing them to a shell where the previous `COPY` instructions do not. 

Without a shell, what can we do? Switch to the `builder` variant.

== Switch to the Builder Variant

The `builder` variant adds bash and other tools specifically for build steps like this.

Edit the Containerfile and change the `FROM` to include the appropriate tag.
[TIP]
====
The `nano` and `emacs` editors are also available if you feel more comfortable with those
====
[source,bash,role=execute,subs=attributes+]
----
vi ~/flask/Containerfile.hi
----
[source,dockerfile,subs=attributes+]
----
# Stage 1: Builder base image
FROM {hummingbird-registry}/python:3.14-builder #<.>
----
<.> The tagging format for `python` images is <version>-builder


[source,bash,role=execute]
----
podman build -t rhhi-demo:builder -f ~/flask/Containerfile.hi ~/flask
----

The build succeeds.
Run it to confirm the app works:

[source,bash,role=execute]
----
podman run -d --rm --name rhhi-builder -p 8080:8080 rhhi-demo:builder
----

Open link:http://flask-{guid}.{domain}/crypto-demo[*the demo application*^] in your browser and confirm the app is working as expected.

[source,bash,role=execute]
----
podman stop rhhi-builder 
----
Changing existing Containerfiles to use hardened images can be fairly straightforward, as in this case. To help understand the default settings for hardened images, we https://gitlab.com/redhat/hummingbird/containers/-/blob/main/images/python-3-14/hummingbird/default/compat.md[provide tables] that compare them to upstream versions.
This not only includes the properties, but also the reasoning behind any changes.


== What's in the Builder Image?

There are some other differences in the packages installed and the tools available. List the images just built. Note that the hardened image is just over half the size of the UBI based image.

[source,bash,role=execute]
----
podman images my-flasksite
----

.*Expected output:*
----
REPOSITORY              TAG         IMAGE ID      CREATED            SIZE
localhost/my-flasksite  hi          7355f3daefdc  About a minute ago  182 MB
localhost/my-flasksite  ubi         472a6a9ebee8  9 minutes ago       254 MB
----
The new image is smaller, but we've included everything in the `builder` image in our application, including a shell that we won't want in production.

[source,bash,role=execute]
----
podman exec -it flask-demo /bin/sh -c 'ls -al /app'
----


[source,bash,role=execute]
----
podman run --rm rhhi-demo:builder rpm -qa | wc -l
----

[source,bash,role=execute]
----
podman run --rm rhhi-demo:builder rpm -qa
----

The builder variant carries dnf, bash, shadow-utils, and their dependencies.
These are needed at build time to *install* Flask, but they are not needed to *run* Flask.
Shipping this image to production is about the equivalent of shipping a slightly smaller version of our UBI image.  
We get faster CVE resolution, but the size and attack surface is still larger than necessary.  
How can we ship just what we need for the runtime image to work in production?

== The Multi-Stage Solution

Multi-stage builds allow you to separate the final application artifacts from build time components and to choose the smallest runtimes for production. 
This keeps with the overall goal of minimal images, reduced attack and risk surfaces. 
Multi-stage builds provide a good blend of flexibility and control.

This will combine the use of different variants for different purposes.

`Containerfile.hardened` separates the two concerns.
Stage one uses the builder variant to install Flask into a prefix directory.
The standard RUN command works as expected since the `builder` includes a shell.
Stage two copies only that prefix into the lean default runtime — the builder and its packages are discarded.

TIP: This is an extremely simple example for the purposes of this lab, production multi-stage builds can be much more complicated to increase isolation between build and runtime dependencies, including full compile chains that only copy final binaries.

[source,bash,role=execute]
----
cat ~/flask/Containerfile.hardened
----

[source,dockerfile]
----
include::example$setup-files/flask/Containerfile.hardened[]
----

Building this image, you'll see that `podman` creates a new image each for each set of steps, delimited by the new `FROM` directive.
The `AS` in that directive let's us refer to those images in later stages.

[source,bash,role=execute]
----
podman build -t rhhi-demo:v1 -f ~/flask/Containerfile.hardened ~/flask
----

== Run the Hardened Image

[source,bash,role=execute]
----
podman run -d --name rhhi-hardened -p 8080:8080 rhhi-demo:v1
----

Open link:http://flask-{guid}.{domain}/crypto-demo[the crypto demo^].
The app behaves identically — same hashes, same form.
The difference is what's under the app.

[source,bash,role=execute]
----
podman stop rhhi-hardened && podman rm rhhi-hardened
----

== Compare all our images

[source,bash,role=execute]
----
podman run --rm rhhi-demo:ubi rpm -qa | wc -l
podman run --rm rhhi-demo:builder rpm -qa | wc -l
podman run --rm rhhi-demo:v1 rpm -qa | wc -l
----

The multi-stage runtime carries only what the interpreter needs at runtime.
Build tools are in stage one and were never committed to the final image.
