= Multi-Stage Hardened Build

Red Hat Hardened Images for Python come in variants.
The `default` variant is distroless — no shell, no package manager, only what the interpreter needs at runtime.
The `builder` variant adds bash, dnf, and shadow-utils for dependency installation.

Starting with the default variant shows what distroless means in practice.

== Try the Default Variant

Write a single-stage Containerfile using the default `hi/python:3.12` runtime:

[source,bash,role=execute]
----
cat > ~/flask/Containerfile.builder << 'EOF'
FROM registry.access.redhat.com/hi/python:3.12
USER root
RUN pip install flask
COPY app.py /app/app.py
WORKDIR /app
EXPOSE 8080
CMD ["python3", "app.py"]
EOF
----

[source,bash,role=execute]
----
podman build -t rhhi-demo:builder -f ~/flask/Containerfile.builder ~/flask
----

.*Expected output (partial):*
----
error running container: ... executable file `/bin/sh` not found: No such file or directory
----

The `RUN` instruction passes commands through a shell.
`hi/python:3.12` has no shell — that is what distroless means.

== Switch to the Builder Variant

The `builder` variant adds bash specifically for build steps like this.
Update the FROM line:

[source,bash,role=execute]
----
sed -i 's/python:3.12$/python:3.12-builder/' ~/flask/Containerfile.builder
----

[source,bash,role=execute]
----
podman build -t rhhi-demo:builder -f ~/flask/Containerfile.builder ~/flask
----

The build succeeds.
Run it to confirm the app works:

[source,bash,role=execute]
----
podman run -d --name rhhi-builder -p 8080:8080 rhhi-demo:builder
----

Open link:http://flask-{guid}.{domain}/crypto-demo[the crypto demo^] in your browser and confirm the app is running.

[source,bash,role=execute]
----
podman stop rhhi-builder && podman rm rhhi-builder
----

== What's in the Builder Image?

[source,bash,role=execute]
----
podman run --rm rhhi-demo:builder rpm -qa | wc -l
----

[source,bash,role=execute]
----
podman run --rm rhhi-demo:builder rpm -qa
----

The builder variant carries dnf, bash, shadow-utils, and their dependencies.
These are needed at build time to install Flask.
They are not needed to run Flask.
Shipping this image to production means shipping a package manager and a shell.

== The Multi-Stage Solution

`Containerfile.hardened` separates the two concerns.
Stage one uses the builder variant to install Flask into a prefix directory.
Stage two copies only that prefix into the lean default runtime — the builder and its packages are discarded.

[source,bash,role=execute]
----
cat ~/flask/Containerfile.hardened
----

[source,bash,role=execute]
----
podman build -t rhhi-demo:v1 -f ~/flask/Containerfile.hardened ~/flask
----

== Compare Package Counts

[source,bash,role=execute]
----
podman run --rm rhhi-demo:ubi rpm -qa | wc -l
podman run --rm rhhi-demo:builder rpm -qa | wc -l
podman run --rm rhhi-demo:v1 rpm -qa | wc -l
----

The multi-stage runtime carries only what the interpreter needs at runtime.
Build tools are in stage one and were never committed to the final image.

== Run the Hardened Image

[source,bash,role=execute]
----
podman run -d --name rhhi-hardened -p 8080:8080 rhhi-demo:v1
----

Open link:http://flask-{guid}.{domain}/crypto-demo[the crypto demo^].
The app behaves identically — same hashes, same form.
The difference is what's under the app.

[source,bash,role=execute]
----
podman stop rhhi-hardened && podman rm rhhi-hardened
----
